# 照片辨識快速更新流程 - 測試文檔

## 📝 新功能說明

使用者傳照片後，可以只輸入「地點 + 數量」即可快速更新庫存，不需要再輸入品項名稱。

---

## 🔄 完整流程

### 流程 A: 照片辨識 → 快速更新

```
1. 使用者傳送產品照片
   ↓
2. Bot 辨識品項並回覆：
   "我看了這張照片，辨識結果：
    
    辨識結果：尿布
    信心：高
    建議品項名稱：尿布
    建議操作：新增 大順家 尿布 1
    
    ━━━━━━━━━━━━━━━━
    ✅ 已記住品項：尿布
    
    請直接回覆「地點 + 數量」即可新增庫存：
    例如：
    • 大順家 1
    • 南屏 0.5
    • 外出用品 2
    
    或者完整輸入：新增 大順家 尿布 1"
   ↓
3. 使用者輸入：大順家 1
   ↓
4. Bot 自動更新庫存並回覆：
   "✅ 使用辨識品項：尿布
    
    已新增 大順家 的 尿布 1包
    目前數量：3.5包"
   ↓
5. pending 狀態已清除，可以繼續其他操作
```

### 流程 B: 辨識失敗 → 完整輸入

```
1. 使用者傳送照片
   ↓
2. Bot 無法從辨識結果提取品項名稱
   ↓
3. Bot 回覆傳統提示：
   "我看了這張照片，先幫你做初步辨識：
    ...
    如果辨識正確，你可以直接回覆像這樣：
    新增 大順家 尿布 1
    ..."
   ↓
4. 使用者需要輸入完整命令
```

### 流程 C: 格式錯誤 → 提示重試

```
1. 使用者傳送照片
   ↓
2. Bot 辨識並記住品項：尿布
   ↓
3. 使用者輸入：大順家（忘記輸入數量）
   ↓
4. Bot 回覆格式錯誤提示：
   "⚠️ 格式不正確
    
    目前記住的品項：尿布
    
    請輸入「地點 + 數量」，例如：
    • 大順家 1
    • 南屏 0.5
    • 外出用品 2
    
    或輸入完整命令：新增 大順家 尿布 1"
   ↓
5. 使用者可以重新輸入（pending 狀態保留）
```

### 流程 D: 取消 pending 狀態

```
1. 使用者傳送照片
   ↓
2. Bot 辨識並記住品項
   ↓
3. 使用者改變主意，輸入其他指令：
   - /start
   - 查看庫存
   - 購買建議
   - 完整的自然語言命令（如：大順新增濕紙巾1）
   ↓
4. Bot 清除 pending 狀態並執行對應指令
```

## 🔍 技術細節

### Pending State 結構

```python
pending_items = {
    chat_id: {
        "item": "品項名稱",
        "timestamp": 1234567890.123
    }
}
```

- **chat_id**: Telegram chat ID (整數)
- **item**: 從 vision API 提取的品項名稱
- **timestamp**: 辨識時間（目前未使用，但保留供未來擴展）

### 品項名稱提取邏輯

從 vision API 回應中尋找包含「建議品項名稱」或「品項名稱」的行：

```
辨識結果：尿布
信心：高
建議品項名稱：尿布  ← 提取這一行
建議操作：新增 大順家 尿布 1
```

提取方法：
1. 按行分割
2. 找到包含「建議品項名稱」的行
3. 以「：」分割，取第二部分
4. 去除空白

### 地點 + 數量解析

**支援格式：**
- `大順家 1` → (大順家, 1.0)
- `大順 2` → (大順家, 2.0)
- `南屏 0.5` → (南屏家, 0.5)
- `南屏家半` → (南屏家, 0.5)
- `外出用品 3` → (外出用品, 3.0)
- `外出 1包` → (外出用品, 1.0)

**正則表達式：**
```python
r'^(大順家|大順|南屏家|南屏|外出用品|外出)\s*([0-9]+(?:\.[0-9]+)?|半)$'
```

### Pending State 清除時機

**自動清除：**
1. 成功執行「地點 + 數量」更新後
2. 使用者輸入系統指令（/start, 查看庫存等）
3. 使用者輸入完整的自然語言命令
4. 使用者再次傳送照片（覆蓋）

**保留 pending：**
1. 格式錯誤時（讓使用者重試）

## ⚠️ 注意事項

### 1. 記憶體暫存

- pending_items 儲存在記憶體中
- Bot 重啟會清空所有 pending 狀態
- 這是最小版本的設計，符合需求

### 2. 品項名稱依賴 Vision API

- 如果 Vision API 回應格式改變，提取可能失敗
- 失敗時會退回到原始流程（要求完整輸入）
- 不會影響系統運作

### 3. 預設動作為「新增」

- 照片辨識後的快速更新，預設動作為「新增」
- 如果要「使用」，需要輸入完整命令

### 4. 多使用者支援

- 使用 chat_id 作為 key，支援多使用者
- 每個使用者的 pending 狀態獨立

### 5. 不影響 inventory.json

- 不需要修改 inventory.json 結構
- 所有資料結構保持不變

## 🎯 設計決策

### 為什麼不使用資料庫？

- 需求明確要求「最小版本，不要加入資料庫」
- 記憶體暫存對於這個使用場景足夠
- Bot 重啟後使用者可以重新傳照片

### 為什麼保留 pending 在格式錯誤時？

- 讓使用者可以重試，不需要重新傳照片
- 如果使用者想取消，可以輸入其他指令

### 為什麼提取品項名稱？

- Vision API 已經建議了品項名稱
- 減少使用者輸入，提升體驗
- 失敗時退回原始流程，不影響功能
